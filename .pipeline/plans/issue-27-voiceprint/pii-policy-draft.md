# 声紋（Voiceprint）生体PIIガバナンス方針 — 承認ドラフト

> **注記**: 本書は Issue #27（Phase 4: 声紋登録による自動命名）の実装着手を止めている
> 承認ブロッカーを解消するための**ドラフト**であり、法務レビューではない。
> 一社内・単一テナント（カボス／日本国内の社内会議利用）を前提に、実務判断として
> 承認可能な粒度で書いている。法的リスクが高いと判断される場合は法務確認を挟むこと。
>
> 関連: `gh issue view 27` / `.pipeline/plans/speaker-attribution-voiceprint/plan.md`
> （Phase 4, Preconditions #4） / `.pipeline/evidence/speaker-attribution-voiceprint/soniox-capability-research.md`

## 1. 適用範囲と法的前提

- 声紋（話者埋め込みベクトル）は**生体PII**である。Soniox は声紋登録・会議横断の
  本人識別機能を持たない（調査済・確定）。本機能は**別の埋め込みアダプタ**（例:
  SpeechBrain/ECAPA、pyannote、または外部ホスト型API）を新設して実現する。
- **本デプロイに直接適用される法体系**: 個人情報保護法（日本）。声紋は
  「個人識別符号」に該当する（個人情報保護法施行令第1条第1号ロ：特定の個人の
  身体の一部の特徴を電子計算機の用に供するために変換した符号であって、
  本人を認証できるもの）。よって声紋データの取得・保存・利用には
  **本人の同意・利用目的の特定・第三者提供の制限**（法17条・18条・27条相当）が適用される。
- プランに引用されている California Civil Code §1798.140（CCPA/CPRA の biometric
  information 定義）は**本デプロイの管轄法ではない**（対象は日本国内・カボス社内利用）。
  ただし要求水準（specific・informed・affirmative consent）が個人情報保護法の実務より
  明確であるため、**両者のうち厳格な方を採用**する: 適用法は個人情報保護法、
  同意・説明の実装水準は CCPA 相当（用途別・明示的・記録付き同意）に合わせる。
- 将来、海外拠点/海外従業員の会議を扱う場合は本節を見直すこと（OPEN DECISION 参照不要、
  現状は日本国内限定のため対象外と明記）。

## 2. 同意（Consent）

- **同意主体**: 声紋を登録される本人（enrolled person）のみ。会議の主催者や
  管理者が代理で同意することはできない。
- **同意のタイミング**:
  - 明示登録フロー: 「声を登録する」ボタンから開始する場合、登録前に同意画面を表示し、
    同意なしに埋め込み抽出・保存を行わない。
  - 暗黙登録（クラスタ命名時の「この声を◯◯として登録」提案）: 提案を**受諾した時点**が
    同意取得のタイミング。UIのデフォルトは非選択（オプトイン）とし、命名操作それ自体を
    同意とみなさない（命名 ≠ 同意）。
  - 本人以外（他者がクラスタを命名して登録を提案するケース）は、登録対象本人への
    別チャネルでの同意確認を必須とする（本人がその場にいない可能性があるため）。
- **同意レコードのスキーマ**（最小項目）:
  `subject_id`（登録対象の profile_id）, `scope`（用途: "会議内自動命名" 固定、
  目的外利用不可）, `method`（"explicit_enroll" | "implicit_suggest_accept"）,
  `consented_at`（timestamp）, `consented_by`（本人 user_id）, `revoked_at`（nullable）。
- **未登録の会議参加者の扱い（OPEN DECISION Bで選択）**: 声紋照合はクラスタ埋め込みを
  登録済プロファイルと比較する処理であり、**未登録者からの埋め込み抽出自体は
  照合目的でのみ一時的に発生**する。方針として
  「**照合のためだけに埋め込みを生成し、登録済プロファイルとの比較後は即時破棄
  （保存しない）**」を採用する。未登録者の埋め込みをディスクやDBに残さないことで、
  同意のない者の生体情報を保持しないことを保証する。

## 3. 保持（Retention）

- **声紋（埋め込みベクトル）の保持期間**: OPEN DECISION A（12/24/36ヶ月）。
  最後の一致（match）または再登録から起算し、期間超過で自動失効・削除する
  バッチジョブを設ける。
- **元音源との紐付け**: 声紋は元のクラスタ音声クリップへの参照を保持しない
  （embedding のみを保存し、生成元の音声ファイルは会議の既存の保持ポリシーに従う）。
  声紋テーブルに `source_meeting_id` は分析目的で残すが、音声バイナリそのものへの
  外部キーは持たせない。
- **再登録**: 同一人物が声紋を更新したい場合（声質変化・誤登録修正）、旧
  embedding を無効化し新規 embedding に切替える。旧 embedding は即時削除（保持しない）。
- **休眠プロファイル**: 保持期間内に一度もマッチしなかったプロファイルは、期間終了時に
  本人へ通知の上、自動削除する。

## 4. 削除／オプトアウト

- **削除APIのカスケード**: `profile 削除 → 紐づく全 voiceprints 削除 →
  audit_log に削除イベントを記録`。この順序と全件削除を保証する（部分削除の
  中途半端な状態を残さない）。
- **SLA**: 削除リクエスト受理から **72時間以内**に完了。完了後は照合が
  当該人物について常に「要確認」に戻る（誤って古い embedding にマッチしない）。
- **既にその声紋で命名済みの過去の文字起こし**: 表示名（テキスト）はそのまま残す。
  声紋（生体データ）のみを削除する。
  - 根拠: 過去の文字起こしに残る「氏名テキスト」はその会議の記録として
    もともと保持されるべき議事情報であり、生体情報そのものではない。
    生体PIIとして削除義務があるのは「声からその人物を識別できる変換符号
    （embedding）」であって、既に人間が確認・記録した名前のテキストではない。
  - 削除後に同じ会議を再文字起こし（`mode="replace"`）した場合は、声紋による
    自動命名は行われず「要確認」に落ちる（過去の名前をテキストとして遡って
    書き換えることはしない）。

## 5. 照合しきい値と人間レビュー

- ロールアウトは計画どおり **suggest → 人間確認 → auto** の3段階。初期リリースは
  常に suggest（人間確認必須）とし、auto段階への移行は個別フラグ・個別承認を要する。
- **初期しきい値**: コサイン類似度で**保守的な値から開始**し、誤マッチ率（false-match
  rate）を実測してから調整する。具体的な初期値は Phase 4 実装時にサンプル会議で
  ROC的な検証を行い決定する（本書では「未検証のまま低いしきい値で auto適用しない」
  ことのみを承認事項として固定する）。
- **しきい値未満**: 「要確認」ラベルを付与し、既存の手動命名フローに委ねる。
  自動的に別人物へ誤って割り当てない。
- **auto段階への移行条件**（承認欄参照）: 一定件数以上の suggest→人間確認の実績で
  誤マッチ率が許容水準以下であることを確認してから有効化する。

## 6. セキュリティ

- **テナント分離**: `speaker_profiles` / `voiceprints` は厳格にテナントスコープ。
  クロステナントの類似度検索を行わない（DBクエリ・インデックスの両方でテナントID
  フィルタを必須化）。
- **暗号化**: embedding は保存時に暗号化する（平文カラムに保存しない）。
  鍵管理は既存のシークレット管理方式に従う。
- **監査ログの必須イベント**: `enroll`（登録）, `match`（照合実行、結果とスコア）,
  `suggest`（サジェスト提示）, `confirm`（人間確認による確定）, `delete`（削除/
  オプトアウト）。各イベントに `subject_id`, `actor_user_id`, `timestamp` を記録する。

## 7. 承認欄

以下を項目ごとに承認/差戻しで判断してください。

- [ ] 1. 適用法は個人情報保護法、実装水準はCCPA相当の厳格な同意基準を採用する
- [ ] 2. 同意は本人単位・オプトイン・目的限定（会議内自動命名のみ）で記録する
- [ ] 3. 未登録者の埋め込みは照合後即時破棄し保存しない（OPEN DECISION B: 案A採用）
- [ ] 4. 声紋の保持期間ポリシー（OPEN DECISION A で選択した期間）を適用する
- [ ] 5. 削除APIは profile→voiceprints→audit_log の順でカスケード削除、SLA 72時間
- [ ] 6. 削除後も過去の文字起こしの名前テキストは残し、生体データのみ削除する
- [ ] 7. 初期リリースは suggest→人間確認必須、auto適用は誤マッチ率実測後に個別承認
- [ ] 8. embedding は暗号化保存、テナント厳格分離、6種の監査ログイベントを実装する

### OPEN DECISIONS（承認者が選択）

- **OPEN DECISION A — 声紋の保持期間**: 12ヶ月 / 24ヶ月 / 36ヶ月 のいずれか
  （最終マッチまたは再登録から起算）。
- **OPEN DECISION B — 未登録参加者の扱い**:
  - 案A（本書の推奨）: 照合のためだけに一時的に embedding を生成し、比較後は
    即時破棄（保存しない）。
  - 案B: 未登録クラスタに対しては embedding 抽出自体を行わない（照合精度は
    下がるが生体データ生成そのものを避ける）。
- **OPEN DECISION C — auto段階への移行基準値**: 何件のsuggest→確認実績、
  何%以下の誤マッチ率で auto を有効化するか（具体的な数値は Phase 4 実装時の
  実測データを見て決定するか、事前に基準値を固定するか）。
